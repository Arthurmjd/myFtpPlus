# 远程文件下载系统需求设计文档

> 项目名称：远程文件下载系统 FDS  
> 项目类型：课程设计  
> 开发语言：C++  
> 通信协议：TCP  
> 系统结构：客户端 / 服务端  
> 核心技术：`select()` I/O、多线程、图形界面、自定义应用层协议、断点续传、权限控制、哈希校验

---

## 1. 项目概述

本项目设计并实现一个基于 TCP 协议的远程文件下载程序，系统采用 C/S 架构，由客户端和服务端组成，支持多客户端并发访问。服务端使用 `select()` I/O 模型，并结合多线程处理文件传输等耗时任务，实现类似 FTP 的目录浏览、文件上传、文件下载和权限管理功能。

在原有课程要求基础上，系统增加以下功能：

- 用户名密码登录与匿名访问
- 常用连接保存
- 远程目录树 / 列表浏览
- 上传、下载、删除、重命名、创建目录
- 上传 / 下载断点续传
- 文件传输完整性哈希校验
- 多任务队列、暂停、继续、取消
- 服务端自带管理员图形面板
- 用户权限与目录级访问控制
- 操作日志审计

### 课程要求对应

1. 有图形化界面，传输协议使用 TCP。
2. 程序由服务端、客户端组成，支持多客户端。
3. 服务端使用 `SELECT I/O` 模型，包含多线程技术，实现目录浏览、文件上传、下载等类似 FTP 程序功能。

## 2. 系统组成

## 2.1 客户端

客户端提供图形界面，主要负责：

- 输入服务器地址和端口
- 选择用户名密码登录或匿名访问
- 保存常用连接
- 浏览远程目录
- 发起上传、下载及其他文件操作
- 展示任务进度、速度和剩余时间

## 2.2 服务端

服务端负责：

- 监听客户端连接
- 认证用户身份
- 校验目录和权限
- 处理目录浏览、上传、下载、删除、重命名、建目录请求
- 支持断点续传与哈希校验
- 记录操作日志

## 2.3 管理员端说明

管理员端不单独作为第三个独立程序存在。  
**管理员端就是服务端自带的管理面板。**

也就是说：

- 运行服务端程序时，同时启动管理员管理界面
- 管理员通过服务端界面完成用户管理、权限管理、目录管理和日志查看

## 3. 功能需求

## 3.1 客户端功能

### 3.1.1 连接与登录

客户端应支持：

- 输入服务器地址
- 输入端口
- 用户名密码登录
- 匿名访问
- 保存常用连接

### 3.1.2 远程目录浏览

登录成功后，客户端以树状或列表形式展示远程目录，支持：

- 刷新
- 路径跳转
- 前进
- 后退
- 返回上级目录
- 显示文件大小、日期、类型等信息

### 3.1.3 文件与目录操作

客户端应支持以下远程操作：

- 上传文件
- 上传文件夹
- 下载文件
- 删除文件
- 删除目录
- 重命名文件
- 重命名目录
- 创建新目录

### 3.1.4 压缩包支持

客户端应明确支持压缩包传输，包括但不限于：

- `.zip`
- `.rar`
- `.7z`
- `.tar`
- `.gz`

### 3.1.5 传输任务显示

所有上传和下载任务都应实时显示：

- 进度条
- 当前速度
- 剩余时间
- 任务状态

### 3.1.6 断点续传

客户端需要支持断点续传：

- 下载时：检测本地已下载部分，从中断位置继续下载
- 上传时：根据服务端记录的偏移量，从中断位置继续上传

### 3.1.7 哈希完整性校验

每次传输完成后，客户端需进行完整性校验：

- 传输前计算源文件哈希
- 传输后获取目标文件哈希
- 对比结果
- 校验失败时提示重传

建议统一使用：

- `SHA-256`

### 3.1.8 多任务队列

客户端支持同时发起多个传输任务，所有任务通过队列管理，每个任务支持：

- 暂停
- 继续
- 取消

---

## 3.2 服务端功能

### 3.2.1 连接与并发

服务端启动后监听默认端口，可支持多个客户端同时连接。

服务端采用：

- `select()` 主线程处理监听与普通请求
- 多线程处理文件传输和耗时任务

### 3.2.2 用户认证

服务端支持：

- 普通用户登录
- 匿名访问

登录成功后为用户分配对应权限和可访问目录范围。

### 3.2.3 管理员管理面板

服务端程序启动后自动运行管理员面板。管理员可以：

- 登录管理员账户
- 添加用户
- 删除用户
- 修改密码
- 启用或停用账户
- 修改用户主目录
- 配置目录级权限
- 查看操作日志

默认管理员账号：

- 用户名：`admin`

密码可首次设定。

### 3.2.4 权限控制

每个用户可被分配以下权限：

- 读：浏览、下载
- 写：上传、创建目录
- 删除
- 重命名

权限可精确到目录级别。

### 3.2.5 目录结构

系统预设目录结构如下：

```text
server_root/
├── public/      公共目录，所有用户和匿名用户可读
├── upload/      上传目录，可写，不允许读取他人文件
├── download/    下载目录，只读
└── users/       用户私有目录
```

规则如下：

- 每个用户拥有独立主目录
- 匿名用户默认仅能访问 `public` 和 `download`
- 管理员可修改用户主目录与可访问范围

### 3.2.6 文件传输与校验

服务端负责：

- 处理上传和下载请求
- 记录断点偏移量
- 支持断点续传
- 传输完成后主动计算哈希
- 返回哈希结果给客户端进行比对

### 3.2.7 日志审计

服务端记录以下事件：

- 登录成功 / 失败
- 上传 / 下载
- 删除
- 重命名
- 创建目录
- 权限拒绝
- 管理员操作

日志至少包含：

- 时间戳
- 用户名
- 事件类型
- 事件详情

## 4. 关键设计要求

## 4.1 通信协议

底层使用 TCP 协议。  
应用层使用自定义消息格式，统一采用：

```text
消息头 + 消息体
```

消息头至少包含：

- 命令字
- 数据长度
- 序号
- 会话标识

这样可以解决 TCP 粘包和拆包问题。

## 4.2 服务端 I/O 模型

服务端主线程必须使用 `select()` I/O 模型：

- 监听新连接
- 监听普通命令
- 管理多个客户端会话

文件传输等耗时操作由工作线程负责。

## 4.3 多线程协作

基本原则：

- `select()` 主线程不直接处理大文件传输
- 上传、下载、哈希计算由工作线程执行
- 每个会话独立运行，保证并发性

## 4.4 大文件支持

系统必须支持大文件传输。  
**单文件传输能力至少支持 2 GB。**

因此设计中需要满足：

- 文件大小字段使用 64 位整数
- 偏移量使用 64 位整数
- 断点续传使用 64 位偏移量
- 读写接口支持超过 2 GB 文件

## 4.5 断点续传

上传和下载都需要保存当前传输偏移量，网络中断后可继续传输。

### 下载续传

- 客户端检测本地已下载文件大小
- 向服务端请求从该偏移量继续下载

### 上传续传

- 服务端记录已接收大小
- 客户端重新上传时先查询当前偏移量
- 从偏移量位置继续发送

## 4.6 哈希校验

系统需要对传输结果进行完整性校验，推荐使用：

- `SHA-256`

校验流程：

1. 传输前计算源文件哈希
2. 传输后计算目标文件哈希
3. 客户端比较两者
4. 若不一致则提示重传

## 6. 非功能需求

| 编号 | 类型 | 要求 |
|---|---|---|
| NFR-01 | 性能 | 局域网环境下文件传输速度稳定 |
| NFR-02 | 并发 | 支持不少于 10 个客户端同时连接 |
| NFR-03 | 稳定性 | 多客户端同时上传/下载时服务端不崩溃 |
| NFR-04 | 可用性 | GUI 操作清晰，状态反馈及时 |
| NFR-05 | 安全性 | 禁止目录穿越和越权访问 |
| NFR-06 | 可恢复性 | 支持上传和下载断点续传 |
| NFR-07 | 一致性 | 文件传输完成后进行哈希校验 |
| NFR-08 | 可扩展性 | 协议和功能模块便于后续扩展 |

最终系统特点如下：

- 图形化客户端
- TCP 协议通信
- 服务端 `select()` I/O 模型
- 多线程并发处理
- 多客户端支持
