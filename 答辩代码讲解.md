# myFtpPlus 答辩代码讲解

这份文档不是给开发者交接用的，而是给答辩时“讲关键功能实现”用的。目标是两件事：

1. 让你知道每个核心功能是怎么实现的。
2. 让你在老师追问时，能说出“为什么这样设计、代码在哪、函数参数是什么意思”。

## 一页总览

这个项目本质上是一个基于 `C++20 + Winsock TCP + Win32 GUI` 的远程文件传输系统，包含服务端和客户端两部分：

- 服务端负责监听端口、管理用户、校验权限、处理目录操作、处理上传下载。
- 客户端负责登录、浏览远程目录、创建目录、删除、重命名、上传、下载、暂停、继续、取消。
- 两端之间不是用现成 FTP 协议，而是自己定义了一套轻量级应用层协议。
- 服务端的控制命令使用 `select()` 主循环处理，上传/下载则交给独立工作线程。
- 客户端界面和服务端界面都使用原生 Win32 控件和消息驱动模型。
- 用户数据持久化到 SQLite，首次无数据库时可导入旧版 TSV 或直接生成种子用户。

如果你只有 1 分钟，可以这样概括：

> 我的项目采用的是“自定义协议 + TCP 传输 + Win32 图形界面 + select 主循环 + 传输线程”的设计。控制命令和数据传输分开处理，控制命令走统一会话，上传下载则单独开连接，支持断点续传和 SHA-256 校验。权限模型使用虚拟路径和最长前缀匹配规则，用户信息持久化到 SQLite，界面层使用 Win32 原生控件和消息分发实现。

## 项目分层

可以把项目理解为四层。

| 层次 | 主要文件 | 作用 |
|---|---|---|
| 公共协议与工具层 | `src/shared/common.hpp`、`src/shared/common.cpp` | 定义协议、封包解包、路径映射、权限规则、哈希、目录枚举等公共能力 |
| 服务端核心层 | `src/server/core.hpp`、`src/server/core.cpp` | 监听端口、维护会话、处理命令、分发上传下载线程 |
| 客户端业务层 | `src/client/protocol.hpp`、`src/client/protocol.cpp`、`src/client/state.hpp`、`src/client/state.cpp` | 客户端控制连接、上传下载逻辑、任务管理、收藏连接、浏览历史 |
| 图形界面层 | `src/client/window.*`、`src/server/window.*`、`src/platform/win32_util.*` | Win32 控件创建、消息循环、界面操作、文件选择对话框 |

如果老师问“这是什么架构”，比较稳妥的回答是：

> 这不是严格意义上的 MVC 或 MVVM，而是一个轻量级分层架构。界面层只负责显示和消息分发，业务逻辑在 `CommandClient`、`TaskManager`、`ServerCore` 里，协议和通用工具统一放在 `shared` 层。

## 自定义协议是怎么设计的

### 1. 协议为什么要自定义

因为本项目的目标不是做标准 FTP 客户端，而是做一个可控、可讲解、适合作业答辩的远程文件系统。自定义协议有三个好处：

- 协议字段完全可控，方便加入 `session`、`seq`、权限相关语义。
- 上传、下载、列表、登录可以统一到同一套命令系统里。
- 更方便展示“我自己设计了应用层协议，而不是只会调第三方库”。

### 2. 协议头长什么样

协议头定义在：

- `src/shared/common.hpp:32`
- `src/shared/common.hpp:68`

核心常量和结构：

- `kMagic = 0x31445346`，表示这是本项目的协议包，注释里写的是 `"FDS1"`。
- `kVersion = 1`，表示协议版本。
- `PacketHeader` 是固定长度包头。

`PacketHeader` 字段含义如下：

| 字段 | 含义 | 作用 |
|---|---|---|
| `magic` | 魔数 | 判断收到的是不是本协议的数据包 |
| `version` | 协议版本 | 为以后协议升级预留 |
| `cmd` | 命令字 | 表示这是登录、列目录、上传数据还是下载结束等 |
| `seq` | 序号 | 用于请求与响应配对 |
| `session` | 会话 ID | 登录成功后，后续请求都绑定到这个用户会话 |
| `length` | 正文长度 | 解决 TCP 粘包/拆包问题 |

一句话解释这个设计：

> TCP 只保证字节流可靠，不保证消息边界，所以我自己定义固定包头，把正文长度写进去，这样接收端就能先读包头，再按长度完整读取包体。

### 4. 包体怎么编码

#### 控制命令包体

控制类命令的正文一般是文本键值对，序列化和反序列化在：

- `SerializePairs`：`src/shared/common.cpp:229`
- `ParsePairs`：`src/shared/common.cpp:237`

编码格式是：

```text
key\tvalue\n
```

例如登录请求大致会长这样：

```text
anonymous    0
username     demo
password     demo123
```

这部分底层还做了转义处理，相关函数在：

- `EscapeField`：`src/shared/common.cpp:158`
- `UnescapeField`：`src/shared/common.cpp:173`
- `MakeLine`：`src/shared/common.cpp:200`
- `ParseLines`：`src/shared/common.cpp:212`

#### 目录列表包体

目录列表不是简单的键值对，而是“多行记录”：

- `PWD\t当前目录`
- `E\tname\tpath\tisDir\tsize\tmtime`

服务端组装列表响应在：

- `ServerCore::BuildListResponse`：`src/server/core.cpp:542`

客户端解析列表响应在：

- `CommandClient::List`：`src/client/protocol.cpp:144`

#### 数据传输包体

上传和下载的数据块不是文本，而是直接把二进制文件内容放入 `body`。

例如：

- 上传时，客户端发送 `UploadData`，包体就是文件块二进制数据。
- 下载时，服务端发送 `DownloadData`，包体也是文件块二进制数据。

这也是为什么协议头里必须有 `length`，因为二进制内容不能靠换行去分隔。

## 登录和会话模型

### 1. 登录流程

客户端登录入口：

- `ClientWindow::ConnectServer`：`src/client/window.cpp:596`
- `CommandClient::Connect`：`src/client/protocol.cpp:59`

服务端登录处理：

- `ServerCore::HandleLogin`：`src/server/core.cpp:557`

登录时序可以概括为：

```text
客户端 -> Login(username/password 或 anonymous)
服务端 -> 校验用户 / 生成匿名会话
服务端 -> 生成 session.id
服务端 -> LoginOk(session, home, username, admin)
客户端 -> 保存 session，后续请求都带上它
```

### 2. 会话为什么有必要

因为登录后，服务端需要记住：

- 这是哪个用户
- 它的主目录是什么
- 它是不是管理员
- 它有哪些权限
- 它是不是匿名用户

这些信息保存在：

- `SessionInfo`：`src/shared/common.hpp:100`
- `sessions_` 容器：`src/server/core.hpp:97`

### 3. 匿名登录如何实现

匿名登录逻辑在：

- `ServerCore::HandleLogin`：`src/server/core.cpp:559`
- `AnonymousRules()`：`src/server/core.cpp:53`

匿名用户的特点：

- 用户名直接写成 `"anonymous"`
- 主目录设为 `"/public"`
- 权限规则只允许 `/public` 和 `/download` 的读权限

所以答辩时可以说：

> 匿名登录不是绕过权限，而是服务端专门生成一个受限会话，并给它绑定只读规则。

判断函数：

1. 找出所有能匹配目标路径的规则

### 4. 为什么要用虚拟路径

路径相关核心代码：

- `NormalizeVirtualPath`：`src/shared/common.cpp:392`
- `VirtualToReal`：`src/shared/common.cpp:421`

设计目的：

- 客户端只看到逻辑路径，比如 `/public/demo.txt`
- 服务端把它映射到真实磁盘 `data/server_root/...`
- 这样不会直接暴露真实文件系统结构
- 还可以统一做权限控制和安全检查

### 5. 如何防止目录穿越

关键就在 `NormalizeVirtualPath`：

- 会把 `\` 统一成 `/`
- 自动补前导 `/`
- 忽略空段和 `.`
- 如果发现 `..`，直接返回空字符串，表示非法路径

所以如果老师问“怎么防止用户访问上级目录”，你可以直接回答：

> 我在路径规范化阶段就拦截了 `..`，非法路径会直接变成空，后续权限校验和真实路径映射都不会通过。

### 3. 为什么用 `select()`

因为控制命令都是小包、短操作，适合统一在一个主循环里处理。这样实现简单，适合课程项目答辩。

- 登录、列目录、创建目录、删除、重命名等轻量命令由 `select()` 主循环直接处理
- 上传、下载这种长耗时任务才单独开线程

这是一种“控制连接复用 + 数据传输分流”的混合模型。

## 上传、下载、断点续传是怎么实现的

这是答辩最值得讲的部分。

### 1. 设计思路

控制连接和传输连接分开：

- 控制连接负责登录、列目录、发上传开始命令、发下载开始命令。
- 真正传文件时，会重新建立一个独立 socket。

原因：

- 控制命令不会被大文件传输阻塞
- 每个传输任务都可以独立暂停、取消、续传
- 服务端更容易把传输交给独立线程

### 2. 上传流程

客户端上传实现：

- `UploadFileSync`：`src/client/protocol.cpp:211`

服务端上传实现：

- `UploadWorker`：`src/server/core.cpp:821`

上传时序：

```text
客户端 -> 新建传输 socket
客户端 -> UploadBegin(path, size, sha256)
服务端 -> 检查权限，查看目标文件已有长度
服务端 -> UploadReady(offset)
客户端 -> 从 offset 位置继续读本地文件
客户端 -> 多次发送 UploadData
客户端 -> UploadEnd(sha256)
服务端 -> 计算最终文件 SHA-256
服务端 -> UploadDone(size, sha256)
客户端 -> 比对服务端返回哈希
```

### 3. 断点续传为什么成立

关键点有两个：

#### 上传续传

服务端在 `UploadWorker` 里：

- 如果目标文件已经存在，就取文件大小作为 `offset`
- 见 `src/server/core.cpp:847`

客户端收到 `UploadReady(offset)` 后：

- 把本地文件读指针 `seekg(offset)`

### 4. SHA-256 校验是怎么做的

- `Sha256Bytes`：`src/shared/common.cpp:303`
- `Sha256String`：`src/shared/common.cpp:328`
- `Sha256File`：`src/shared/common.cpp:332`

- 不是 OpenSSL
- 依赖 `-lbcrypt`
- 构建位置：`build.ps1:46`、`build.ps1:64`

上传时：

1. 客户端先算本地文件哈希
2. 发 `UploadBegin` 时带上 `sha256`
3. 上传结束后，服务端重新算目标文件哈希
4. 服务端把最终哈希放在 `UploadDone`
5. 客户端再比对一遍

1. 服务端在 `DownloadDone` 中带上最终文件哈希
2. 客户端接收完成后计算本地下载文件哈希
3. 两边一致才算成功

实现原理：

- `pauseWanted`、`cancelWanted` 是原子标志位
- 进度回调里不断检查它们
- 如果发现要暂停，就返回 `FlowControl::Pause`

> 这里的“暂停”不是强行挂起线程，而是让传输函数主动返回，然后保留已完成偏移，后续再重新建连续传。

这是一个答辩时很加分的点，因为它说明你没有用危险的 `SuspendThread`，而是用了更合理的协作式暂停。

## 3 分钟答辩推荐讲稿

如果老师让你快速讲实现，你可以按这个顺序说：

### 第一步：先讲架构

> 我的项目分成四层：`shared` 负责协议和公共工具，`server` 负责服务端逻辑，`client` 负责客户端业务和任务管理，`platform` 负责 Win32 的界面辅助封装。

### 第二步：讲协议

> 客户端和服务端通过 TCP 通信，但我没有直接发裸字符串，而是设计了固定包头加包体的自定义协议。包头里有 `magic`、`version`、`cmd`、`seq`、`session`、`length`，这样既能解决 TCP 粘包，也能支持登录会话和命令分发。

### 第三步：讲服务端模型

> 服务端控制命令用 `select()` 主循环统一处理，上传下载这种长耗时任务再单独开线程。这样控制操作不会被大文件传输阻塞。

### 第四步：讲权限和安全

> 所有路径都先做规范化，禁止 `..`。权限规则使用 `/path:RWDN` 形式，服务端通过最长前缀匹配判断某个用户是否有读写删改权限。

### 第五步：讲断点续传和校验

> 上传时服务端先告诉客户端目标文件已有多少字节，客户端从这个偏移继续发送；下载时客户端把本地已有长度发给服务端，服务端从这个位置继续发。传输完成后双方都会计算 SHA-256，确保文件内容一致。

### 第六步：讲界面

> 界面层使用 Win32 原生控件和消息驱动模型，所有按钮和列表都是 `CreateWindowW` 创建的。耗时任务不放在界面线程里，所以界面不会卡死。

## 模拟老师提问与参考回答

下面这一节建议你答辩前至少通读两遍。

### 问题 1：你为什么不用现成 FTP，而要自定义协议？

参考回答：

> 因为这个项目重点是展示我对应用层协议、会话、权限和断点续传机制的理解。自定义协议可以更直接地表达登录、目录操作、上传下载、会话 ID 和权限控制这些语义，也更适合课程答辩展示。

### 问题 2：TCP 不是已经可靠传输了吗？为什么还要自己定义包头？

参考回答：

> TCP 只保证字节流可靠，不保证消息边界。也就是说接收端不知道一条完整消息从哪里开始、在哪里结束，所以我增加了固定包头，里面带 `length` 字段，接收端先读头，再按长度读正文，就能解决粘包和拆包问题。

### 问题 3：`magic` 和 `version` 的作用是什么？

参考回答：

> `magic` 用来判断收到的是不是我这个协议的数据，防止误解包。`version` 是协议版本号，后续如果协议升级，可以做兼容判断。

### 问题 4：`seq` 在你的项目里有什么意义？

参考回答：

> `seq` 是请求序号，主要用于请求和响应配对。当前项目里每次控制请求都是一问一答，它的存在是为了让协议更规范，也方便以后扩展成更复杂的并发请求模型。

### 问题 5：登录成功后为什么还要 `session`？

参考回答：

> 因为服务端需要记住当前连接对应的是谁、主目录是什么、是否管理员、有哪些权限。登录后生成 `session.id`，后续每个请求都带这个会话 ID，服务端就能恢复用户上下文，而不是每次都重新传用户名密码。

### 问题 6：你的权限控制是怎么做的？

参考回答：

> 权限规则使用 `/path:RWDN` 这种格式，服务端收到请求后先把路径规范化，再使用最长前缀匹配去找最具体的规则，最后判断是否包含所需权限位。这样既能表达公共目录权限，也能表达用户私有目录权限。

### 问题 7：你怎么防止用户访问到不该访问的上级目录？

参考回答：

> 我在 `NormalizeVirtualPath` 中统一路径格式，并直接拒绝 `..`。也就是说，只要请求路径里有目录穿越意图，规范化就失败，后续不会进入真实路径映射和文件系统操作。

### 问题 8：断点续传具体是怎么实现的？

参考回答：

> 上传时，服务端先检查目标文件已经有多少字节，然后把这个偏移返回给客户端，客户端从这个偏移继续读本地文件发送。下载时，客户端先看本地文件已有多大，再把 offset 发给服务端，服务端从这个位置继续发送剩余数据。

### 问题 9：你的暂停和继续为什么不是直接挂起线程？

参考回答：

> 因为强行挂起线程风险比较高，容易造成资源状态不一致。我采用的是协作式控制：界面只设置 `pauseWanted` 或 `cancelWanted` 标志，传输线程在进度回调里主动检查这些标志并安全退出，之后再依靠断点续传恢复。

### 问题 10：SHA-256 校验放在什么位置？为什么要做两边校验？

参考回答：

> 上传时客户端先计算本地文件哈希，服务端写完后再算目标文件哈希并返回，客户端再比对一遍。下载时服务端在结束时返回文件哈希，客户端下载完再算本地文件哈希比对。这样可以验证传输后文件内容是否一致，而不是只相信字节数。

### 问题 11：为什么服务端不用“一个客户端一个线程”的模型？

参考回答：

> 因为登录、列目录、创建目录这些都是短操作，如果每来一个连接都单独开线程，管理成本会更高。我的做法是控制命令统一走 `select()` 主循环，只有真正长耗时的上传下载才开线程，这样更适合这个项目规模。

### 问题 12：你的界面用了什么模型？

参考回答：

> 用的是原生 Win32 消息驱动模型。窗口通过 `RegisterClassW` 和 `CreateWindowW` 创建，用户操作通过 `WndProc` 接收 `WM_COMMAND`、`WM_NOTIFY`、`WM_TIMER` 等消息，再调用对应业务函数。

### 问题 13：你的客户端为什么不会因为上传下载而卡死？

参考回答：

> 因为界面线程只负责消息循环和界面刷新，上传下载都放在 `TaskManager` 创建的后台线程里执行，界面只是通过定时器刷新任务状态。

### 问题 14：你这个项目算 MVC 吗？

参考回答：

> 严格来说不是完整 MVC 框架，而是轻量分层架构。`ClientWindow` 和 `ServerWindow` 更偏 View + Controller，`ServerCore`、`CommandClient`、`TaskManager` 负责业务逻辑，`shared` 层负责协议和工具。

### 问题 15：为什么用户数据要存 SQLite，而不是纯文本？

参考回答：

> SQLite 比纯文本更适合结构化用户数据管理，扩展性更强，比如以后加更多字段会更方便。同时我保留了旧版 TSV 导入逻辑，兼顾了演示性和兼容性。

### 问题 16：为什么 SQLite 要动态加载 DLL？

参考回答：

> 这样部署更灵活。Windows 某些环境自带 `winsqlite3.dll`，没有的话也可以额外提供 `sqlite3.dll`。相比硬绑定第三方库，这种方式更轻量。

### 问题 17：如果老师问“你的程序里体现了哪些工程化思想”，怎么答？

参考回答：

> 我觉得主要有五点：第一，协议和业务分层；第二，路径规范化和权限校验保证安全；第三，控制连接和传输连接分离；第四，断点续传和哈希校验保证可靠性；第五，使用 RAII 封装 socket、OLE、SQLite statement 等资源，减少泄漏风险。

### 问题 18：你代码里有哪些 RAII 思想？

## 补充：select I/O 模型专项说明

这一节建议你重点看，因为老师非常喜欢问：

- 你这里有没有用 `select`？
- `select` 用在什么地方？
- 它属于什么 I/O 模型？
- 为什么这里用 `select`，而不是每个连接一个线程？

### 1. 这个项目里有没有用 `select`？

有，而且是明确使用了。

核心位置在：

- `ServerCore::Loop`：`src/server/core.cpp:1004`

服务端启动后会创建一个主循环线程：

- `ServerCore::Start`：`src/server/core.cpp:83`
- 启动循环线程：`src/server/core.cpp:97`

这个主循环线程内部就一直在调用：

```cpp
const int ready = select(static_cast<int>(maxSocket + 1), &readSet, nullptr, nullptr, &timeout);
```

对应代码位置：

- `src/server/core.cpp:1028`

### 2. `select` 在这里具体干什么？

它做的是 I/O 多路复用。

简单说，不是为每个控制连接单独开一个线程，而是让一个线程同时监听多个 socket：

- 监听 socket：看看有没有新客户端接入
- 已连接客户端 socket：看看有没有控制命令到达

主循环大致逻辑是：

1. 把监听 socket 放进 `fd_set`
2. 把所有客户端控制连接也放进 `fd_set`
3. 调用 `select()` 等待“哪个 socket 可读”
4. 如果监听 socket 可读，就 `accept()` 新连接
5. 如果某个客户端 socket 可读，就 `RecvPacket()` 读协议包
6. 再按 `cmd` 分发到登录、列目录、创建目录、删除、重命名等处理函数

对应代码片段位置：

- `FD_ZERO / FD_SET`：`src/server/core.cpp:1006`
- 监听 socket 加入集合：`src/server/core.cpp:1010`
- 客户端 socket 加入集合：`src/server/core.cpp:1020`
- `select()` 等待：`src/server/core.cpp:1028`
- `accept()` 新连接：`src/server/core.cpp:1033`
- 遍历可读客户端：`src/server/core.cpp:1049`
- 收包：`src/server/core.cpp:1054`
- 命令分发：`src/server/core.cpp:1062`

### 3. 它属于什么 I/O 模型？

比较标准的答法是：

> 服务端控制连接部分使用的是 `select` I/O 多路复用模型，也可以理解为“单线程事件循环 + 多连接监听”的方式。  
> 但本项目不是纯 Reactor，因为上传下载这种耗时任务没有继续留在事件循环里，而是转交给独立工作线程处理。

如果老师继续问“那你这是阻塞还是非阻塞”，你可以这样答：

> 监听 socket 在创建时被设置成非阻塞，代码在 `CreateListenSocket`；控制主循环通过 `select()` 先判断可读，再读取协议包。整体上，控制连接层面体现的是 `select` 多路复用思想。

相关代码：

- 创建监听 socket：`src/shared/common.cpp:62`
- 设置监听 socket 非阻塞：`src/shared/common.cpp:67`

### 4. 为什么这里要用 `select`？

因为项目里有两类操作：

- 轻量操作：登录、列目录、创建目录、删除、重命名
- 重量操作：上传、下载

轻量操作特点是：

- 数据包小
- 处理时间短
- 很适合放在一个统一主循环里处理

所以我用了 `select()` 来统一处理控制命令，这样好处是：

- 结构清晰
- 比“每个控制连接一个线程”更轻
- 更适合课程设计规模
- 方便答辩时说明“我用了 I/O 多路复用”

### 5. 为什么上传下载不继续用 `select` 处理？

因为上传下载是长耗时、大数据量任务，如果也放在主循环里，会有两个问题：

- 会阻塞其它控制命令响应
- 主循环逻辑会变得很复杂

所以项目采用的是混合模型：

- 控制命令：`select()` 主循环处理
- 文件传输：独立线程处理

对应代码：

- 上传线程分发：`ServerCore::SpawnUploadWorker`，`src/server/core.cpp:778`
- 下载线程分发：`ServerCore::SpawnDownloadWorker`，`src/server/core.cpp:793`
- 上传线程主体：`src/server/core.cpp:821`
- 下载线程主体：`src/server/core.cpp:924`

一句话总结这个设计：

> 我不是把所有网络操作都塞进 `select()`，而是让 `select()` 负责“连接管理和轻量命令调度”，让工作线程负责“大文件传输”。

### 6. 老师如果问“select 具体体现在哪里”，你可以直接这样答

参考回答：

> 有，服务端控制层明确使用了 `select` I/O 多路复用模型。代码在 `ServerCore::Loop`，文件是 `src/server/core.cpp`。这个循环会把监听 socket 和所有客户端控制连接加入 `fd_set`，调用 `select()` 等待可读事件，然后分别处理新连接接入和控制命令收发。

### 7. 老师如果问“你这个 select 是 reactor 吗”，怎么答

参考回答：

> 可以说它有 Reactor 的味道，但不是完整框架化的 Reactor。因为它确实有事件循环、可读事件检测和命令分发，但是上传下载没有继续在事件循环内异步处理，而是转交给独立线程。所以更准确地说，它是“基于 select 的轻量事件循环 + 传输线程”的混合模型。

### 8. 老师如果问“为什么不用 epoll / IOCP”，怎么答

参考回答：

> 这个项目是 Windows 课程设计规模，目标是结构清晰、容易讲解、功能完整。`select()` 更容易实现和答辩展示，也足够支撑这个项目的并发规模。如果追求更高并发，Windows 平台更适合进一步演进到 IOCP。

### 9. 老师如果问“select 的局限性是什么”，怎么答

参考回答：

> `select()` 的局限主要有三点：第一，随着连接数增加，遍历 `fd_set` 的成本会上升；第二，它更适合中小规模连接；第三，接口层面不如更现代的事件机制高效。所以我在项目里只让它处理控制连接，而不是承担整个大文件传输过程。

### 10. 你答辩时关于 select 最稳的一段话

可以直接背这一段：

> 这个项目在服务端控制连接部分使用了 `select` I/O 多路复用模型，核心代码在 `src/server/core.cpp` 的 `ServerCore::Loop`。我把监听 socket 和所有客户端控制连接放进 `fd_set`，通过 `select()` 等待可读事件，再分别处理新连接和控制命令。这样登录、列目录、创建目录、删除、重命名这些轻量操作都能由一个主循环统一调度。而上传下载由于是长耗时、大数据量任务，我没有继续放在 `select` 循环里，而是转交给独立工作线程处理，所以整体上是 `select` 主循环加传输线程的混合模型。

## 补充：select 和其他模型的对比

这一节是专门给答辩追问准备的。老师通常不会只问“你用了什么”，还会问“为什么不用别的”。这里你要讲清楚的是：

- `select` 不是唯一方案
- 我知道它和其他模型的差别
- 我是结合项目规模和课程设计目标做了取舍

### 1. 和阻塞式单连接模型对比

阻塞式单连接模型可以理解成：

- `accept()` 一个客户端
- 然后一直阻塞读这个客户端
- 直到这个客户端结束，再处理下一个

优点：

- 最简单
- 最容易写

缺点：

- 同一时刻几乎只能服务一个连接
- 一个客户端卡住，其他客户端就要等
- 不适合有多个用户同时操作

和本项目对比：

> 我的项目没有采用这种最原始的阻塞式串行处理，而是使用 `select()` 让一个主循环同时关注多个控制连接，所以多个客户端都可以并发发登录、列目录这类控制命令。

### 2. 和“每个连接一个线程”模型对比

这是一种很常见的早期服务器模型：

- 每来一个客户端连接，就创建一个线程
- 线程里一直阻塞处理这个客户端的所有请求

优点：

- 思路直观
- 单个连接代码写起来像同步流程

缺点：

- 连接一多，线程数量会很多
- 线程切换和资源开销大
- 控制连接其实大多数时间是空闲的，给每个控制连接常驻一个线程不划算

和本项目对比：

> 如果我把每个控制连接都做成一个常驻线程，课程项目也能跑，但代价是结构会更重。因为登录、列目录、创建目录这种命令很轻，使用 `select()` 统一调度更合适。  
> 但对于上传下载这种真正耗时的操作，我又没有强行塞回 `select`，而是单独开线程处理，所以我的方案本质上是在“每连接一线程”和“纯多路复用”之间取了一个折中。

### 3. 和 `poll` 模型对比

`poll` 和 `select` 都属于 I/O 多路复用。

相同点：

- 都是一个线程监听多个连接
- 都要先等待事件，再处理可读/可写 socket

不同点：

- `select` 使用 `fd_set`
- `poll` 使用数组/结构体描述符集合
- `poll` 在接口扩展性上通常比 `select` 更自然

在本项目里为什么没用 `poll`：

> 因为这个项目是 Windows 上的课程设计，`select` 更常见、可讲解性更强、实现也更直接。对于当前这个连接规模，`select` 足够完成需求。

### 4. 和 `epoll` 模型对比

`epoll` 是 Linux 下更高效的事件通知机制，适合高并发服务器。

它和 `select` 的主要区别：

- `select` 每次都要重新组织和遍历描述符集合
- `epoll` 更适合大量连接场景
- `epoll` 事件通知机制更高效

但它的问题是：

- 它是 Linux 平台方案
- 本项目运行环境是 Windows

所以答辩时最稳的说法是：

> `epoll` 的确比 `select` 更适合 Linux 高并发服务器，但本项目是 Win32 + Winsock 环境，课程设计目标也不是做上万连接的高并发网关，所以我没有选 `epoll`。

### 5. 和 `IOCP` 模型对比

如果老师比较懂 Windows，很可能会追问：

- 既然你是 Windows，为什么不用 `IOCP`？

这个问题你一定要会答。

`IOCP` 是 Windows 下更高级的高性能异步 I/O 模型。

优点：

- 很适合高并发
- 可扩展性比 `select` 更强
- 在 Windows 服务器开发里很经典

缺点：

- 编程复杂度明显更高
- 对课程设计项目来说，代码量和理解门槛都更大
- 不如 `select` 直观，答辩展示成本也更高

所以这里的推荐答法是：

> 如果目标是工业级 Windows 高并发服务器，`IOCP` 会是更好的选择。  
> 但这个项目的重点是完整实现远程文件管理、权限、断点续传和图形界面，而不是追求极限并发。`select()` 更容易控制复杂度，也更方便把协议、会话和功能实现讲清楚。

### 6. 和纯异步事件驱动模型对比

纯异步事件驱动模型通常会把：

- 连接管理
- 收包
- 发包
- 状态机推进
- 传输过程

全部放进统一事件循环里。

优点：

- 理论上线程数更少
- 高并发场景下吞吐更好

缺点：

- 状态机会变复杂
- 文件传输、暂停、取消、续传这些逻辑会更难讲、更难写

本项目没有这么做，而是：

- 控制命令用事件循环
- 文件传输用线程

这就是一种偏工程化的折中方案。

### 7. 你这个项目最终属于哪一种？

最准确的说法不是只说“`select` 模型”，而是说：

> 我的项目在控制连接部分使用 `select` I/O 多路复用模型，在文件传输部分使用工作线程模型，所以整体上是“`select` 事件循环 + 传输线程”的混合模型。

这句话比单纯说“我用了 `select`”更完整，也更专业。

### 8. 答辩时可以直接背的对比总结

如果老师问“为什么选 `select`，不选其他模型”，你可以直接这样答：

> 我对几种常见模型做过取舍。最简单的阻塞串行模型并发能力太弱；每连接一线程模型实现直观，但控制连接线程会很多；`epoll` 更适合 Linux 高并发；`IOCP` 更适合 Windows 工业级高并发，但复杂度高。  
> 我的项目是 Win32 课程设计，控制命令大多是轻量操作，所以我用 `select()` 统一管理多个控制连接；而上传下载这种长耗时任务再交给独立线程处理。这样实现复杂度、答辩可解释性和功能完整性之间比较平衡。

### 9. 一张表快速记忆

| 模型 | 特点 | 优点 | 缺点 | 本项目是否采用 |
|---|---|---|---|---|
| 阻塞串行 | 一次只处理一个连接 | 最简单 | 并发最差 | 否 |
| 每连接一线程 | 一个连接一个线程 | 逻辑直观 | 线程开销大 | 部分借鉴，仅用于传输任务 |
| `select` | 多路复用，统一监听多个连接 | 结构清晰，适合中小规模 | 扩展性一般 | 是，用于控制连接 |
| `poll` | 和 `select` 类似 | 接口更自然 | 本质上仍要遍历 | 否 |
| `epoll` | Linux 高并发事件机制 | 高效 | 非 Windows 方案 | 否 |
| `IOCP` | Windows 高性能异步 I/O | 并发能力强 | 实现复杂 | 否 |

### 10. 老师可能继续追问的两个问题

## 一、文件上传的完整执行链路

上传不是一个函数单独完成的，而是跨了 4 层：

1. 客户端界面层发起上传请求
2. 客户端任务层创建上传任务
3. 客户端协议层建立传输连接并分块发送
4. 服务端工作线程接收并落盘

### 1. 第一步：用户在客户端点击“上传文件”

- `ClientWindow::UploadFile`：`src/client/window.cpp:649`

1. 先判断当前是否已经登录：`client_.Connected()`
2. 调用文件选择对话框 `win32::PickOpenFile(hwnd_)`
3. 取出本地文件名
4. 调用 `QueueTask(true, file, JoinRemotePath(CurrentPath(), name))`

这里 `true` 的意思是“这是上传任务”，不是下载任务。

这一步非常适合答辩时说：

> 客户端界面层本身不直接发网络包，它只是采集用户输入，然后把上传操作封装成一个任务，交给任务管理器。

### 2. 第二步：把上传操作封装成任务

相关函数：

- `ClientWindow::QueueTask`：`src/client/window.cpp:590`
- `TaskManager::Add`：`src/client/state.cpp:167`

`QueueTask` 会调用：

```cpp

```

这里会构造一个 `TransferTask`，关键字段包括：

- `task->upload = true`
- `task->local = 本地路径`
- `task->remote = 远程路径`
- `task->state = Queued/Running`

任务结构定义在：

- `TransferTask`：`src/client/state.hpp:58`

这个结构里和上传最相关的字段有：

| 字段 | 作用 |
|---|---|
| `upload` | 标识是上传还是下载 |
| `local` | 本地文件路径 |
| `remote` | 远程文件路径 |
| `done` | 当前已传字节数 |
| `total` | 总字节数 |
| `resumeFrom` | 本次任务从哪个偏移开始续传 |
| `pauseWanted` | 用户是否请求暂停 |
| `cancelWanted` | 用户是否请求取消 |
| `state` | 当前任务状态 |

### 3. 第三步：任务管理器启动后台线程

核心函数：

- `TaskManager::Start`：`src/client/state.cpp:222`

这个函数做了三件关键事：

1. 把任务状态设成 `Running`
2. 清空暂停/取消标志
3. 启动一个 `std::thread` 在后台真正执行上传

后台线程里最关键的是这个分支：

```cpp
const TransferResult result =
    task->upload ? UploadFileSync(connection, task->local, task->remote, progress)
                 : DownloadFileSync(connection, task->remote, task->local, progress);
```

也就是说：

- 上传任务走 `UploadFileSync`
- 下载任务走 `DownloadFileSync`

这里的 `progress` 是一个进度回调，它不是装饰性的，而是整个“暂停、取消、显示进度、记录续传偏移”的核心。

### 4. 第四步：进度回调如何驱动任务状态

进度回调在：

- `src/client/state.cpp:233`

逻辑非常关键：

#### 4.1 更新已完成字节数

```cpp
task->done = done;
task->total = total;
```

这意味着任务列表里看到的进度数据，不是另算的，而是传输层每推进一次就回填到任务对象。

#### 4.2 记录本次任务的续传起点

```cpp
if (!startInfoSet) {
    task->resumeFrom = done;
    SetTaskMessage(task, StartInfoText(task->upload, done));
    startInfoSet = true;
}
```

这里很多同学答辩容易说不清：

- `resumeFrom` 不是“总共传了多少”
- 它记录的是“这次任务开始时已经有多少字节”

比如：

- 如果是首次上传，`done = 0`，那就是从头开始
- 如果是续传上传，`done = 1048576`，那说明前 1MB 已经存在，这次是从 1MB 继续

#### 4.3 检查暂停和取消

```cpp
if (task->cancelWanted.load()) {
    return FlowControl::Cancel;
}
if (task->pauseWanted.load()) {
    return FlowControl::Pause;
}
return FlowControl::Continue;
```

这里说明暂停和取消不是操作系统强停线程，而是传输过程每前进一段主动查询控制标志，再决定继续还是退出。

这点答辩时要强调：

> 我采用的是协作式中断，不是粗暴挂起线程。

### 5. 第五步：客户端协议层真正执行上传

核心函数：

- `UploadFileSync`：`src/client/protocol.cpp:211`

这个函数可以拆成 7 个阶段来讲。

## 二、UploadFileSync 代码逻辑逐步讲解

### 阶段 1：读取本地文件大小

代码位置：

- `src/client/protocol.cpp:213`

```cpp
const auto total = std::filesystem::file_size(localPath, ec);
```

这里拿到上传文件总长度 `total`。后面进度显示、服务端总大小记录、续传合法性判断都要用它。

如果文件大小读不到，直接失败：

```cpp
return MakeResult(TransferOutcome::Failed, "本地文件不可读");
```

### 阶段 2：先计算本地文件哈希

代码位置：

- `src/client/protocol.cpp:219`

```cpp
const auto localHash = Sha256File(localPath);
```

这个 `localHash` 有两个作用：

1. 上传开始时告诉服务端，最终文件应该是什么哈希
2. 上传完成后对服务端回传结果再做比对

也就是说，客户端不是盲传，它在传之前就已经知道“正确答案”。

### 阶段 3：打开本地文件，建立传输连接

代码位置：

- 打开本地文件：`src/client/protocol.cpp:224`
- 建立传输 socket：`src/client/protocol.cpp:229`

这里要注意：

> 上传不是复用控制连接传的，而是重新调用 `ConnectSocket(connection.host, connection.port)` 建立新的传输连接。

原因前面说过：

- 避免大文件阻塞控制连接
- 传输任务和控制命令解耦

### 阶段 4：发送 `UploadBegin`

代码位置：

- `src/client/protocol.cpp:234`

发送内容是：

```cpp
SerializePairs({
    {"path", remotePath},
    {"size", std::to_string(total)},
    {"sha256", localHash},
})
```

也就是告诉服务端三件事：

1. 我想上传到哪个远程路径
2. 这个文件总共多大
3. 这个文件最终正确的 SHA-256 应该是多少

这一步发送的是“上传协商请求”，不是文件内容。

### 阶段 5：等待服务端返回 `UploadReady(offset)`

代码位置：

- 等待响应：`src/client/protocol.cpp:245`
- 解析 `offset`：`src/client/protocol.cpp:249`

服务端返回的核心字段是：

```cpp
offset
```

含义是：

> 服务端当前这个目标文件已经有多少字节了，客户端应该从这里继续传。

这就是断点续传的关键协商点。

客户端收到后会做合法性检查：

```cpp
if (offset > total) {
    return MakeResult(TransferOutcome::Failed, "服务器返回的续传偏移异常");
}
```

然后直接：

```cpp
in.seekg(static_cast<std::streamoff>(offset));
```

意思是本地文件读指针直接跳到 `offset`，前面那部分不再重复发送。

### 阶段 6：循环读取本地文件并发送 `UploadData`

代码位置：

- `src/client/protocol.cpp:261`

核心变量：

- `buffer`：每次读取 64KB
- `done`：当前已完成字节数，初始值就是 `offset`

初始化：

```cpp
std::vector<char> buffer(kChunkSize);
std::uint64_t done = offset;
```

这就非常关键：

> `done` 不是从 0 开始，而是从服务端已存在的偏移开始。  
> 所以进度条看到的是完整进度，而不是“本次新增传输进度”。

循环逻辑：

1. 先调用 `onProgress(done, total)` 检查是否暂停/取消
2. 从本地文件读一块到 `buffer`
3. 发一个 `UploadData` 包，包体就是这块二进制数据
4. `done += got`
5. 继续下一轮

发送数据代码：

```cpp
SendPacket(sock.Get(), Cmd::UploadData, 1, connection.session,
           std::string(buffer.data(), static_cast<std::size_t>(got)))
```

这里 `body` 不是文本，是文件块原始字节。

### 阶段 7：发送 `UploadEnd` 并等待 `UploadDone`

代码位置：

- 发送结束：`src/client/protocol.cpp:283`
- 等待完成：`src/client/protocol.cpp:287`

发送结束包时又带了一次 `sha256`：

```cpp
SerializePairs({{"sha256", localHash}})
```

设计上相当于再次声明：

> 我这边认为最终文件的正确哈希是这个。

然后客户端等待服务端返回：

- `UploadDone`

服务端返回里会带：

- `size`
- `sha256`

客户端最终校验：

```cpp
const auto result = ParsePairs(packet.body);
const auto hashIt = result.find("sha256");
if (hashIt == result.end() || hashIt->second != localHash) {
    return MakeResult(TransferOutcome::Failed, "上传后的哈希校验失败");
}
```

这一句是上传最终成功的最后一道门。

## 三、服务端 UploadWorker 是怎么接住上传的

客户端发起上传之后，服务端不会在 `select` 主循环里直接把整个文件收完，而是会把这个连接移交给上传工作线程。

### 1. 主循环收到 `UploadBegin`

代码位置：

- `ServerCore::HandleCommand`：`src/server/core.cpp:738`

当 `req.cmd == Cmd::UploadBegin` 时：

```cpp

keepSocketOpen = true;
SpawnUploadWorker(sock, req);
```

这三行很值得讲：

- `removeSocket = true`：说明这个 socket 不再由控制连接列表管理
- `keepSocketOpen = true`：但是不要关闭，因为工作线程还要继续用
- `SpawnUploadWorker(sock, req)`：把上传任务移交给后台线程

### 2. 创建上传线程

代码位置：

- `ServerCore::SpawnUploadWorker`：`src/server/core.cpp:778`

这里会：

1. 记录这个传输 socket，便于停服时统一中断
2. `std::thread` 启动后台线程
3. 在线程内调用 `UploadWorker(sock, req)`

### 3. UploadWorker 的前置校验

代码位置：

- `ServerCore::UploadWorker`：`src/server/core.cpp:821`

首先做三类校验：

#### 3.1 会话校验

```cpp
auto session = FindSession(req.session);
```

如果会话失效，直接返回错误。

#### 3.2 参数校验

从 `req.body` 中取出：

```cpp
path
```

如果没有 `path`，返回错误。

#### 3.3 权限和路径校验

```cpp
const auto resolved = ResolvePath(*session, pathIt->second, PermWrite, true);
```

这里要求的是 `PermWrite`，因为上传本质上是写操作。

如果权限不够，或者路径非法，都不会进入真正写文件流程。

### 4. 服务端如何计算续传偏移 `offset`

代码位置：

- `src/server/core.cpp:847`

```cpp
std::uint64_t offset =
    std::filesystem::exists(resolved->real, ec)
        ? std::filesystem::file_size(resolved->real, ec)
        : 0;
```

这句是整个断点续传最核心的服务端逻辑。

含义：

- 如果目标文件已经存在，说明之前已经传过一部分
- 那么已有文件大小就是续传起点
- 如果不存在，就从 0 开始

同时还会记录：

```cpp
const std::uint64_t requestedTotal = beginMeta.contains("size") ? ParseU64(beginMeta.at("size")) : 0;
const std::uint64_t total = std::max(requestedTotal, offset);
const std::uint64_t resumeOffset = offset;
```

这里几个变量的意义：

| 变量 | 含义 |
|---|---|
| `offset` | 当前文件已存在的长度，也就是客户端应该从哪继续传 |
| `requestedTotal` | 客户端声明的文件总长度 |
| `total` | 用于显示进度的总长度，至少不能小于 `offset` |
| `resumeOffset` | 纯展示用途，告诉界面这是不是断点续传 |

### 5. 服务端如何写文件

代码位置：

- `src/server/core.cpp:853`

```cpp
std::ofstream out(resolved->real, std::ios::binary | std::ios::app);
```

注意这里用了：

- `std::ios::app`

这就意味着：

> 服务端不是从头覆盖写，而是追加写。  
> 所以只要前面的数据已经存在，新的数据块就会从文件尾继续拼接。

这就是上传续传成立的第二个关键点。

### 6. 服务端把 `offset` 告诉客户端

代码位置：

- `src/server/core.cpp:864`

```cpp

           SerializePairs({{"offset", std::to_string(offset)}}))
```

这一步其实就是上传协议里的“续传协商响应”。

客户端拿到它以后才知道自己从哪开始继续发。

### 7. 服务端如何接收 `UploadData`

代码位置：

- `src/server/core.cpp:876`

后面进入一个 `while (true)` 循环，不断收包：

```cpp
if (!RecvPacket(sock, packet)) { ... }
```

如果收到的是：

```cpp
packet.cmd == Cmd::UploadData
```

就执行：

```cpp

offset += static_cast<std::uint64_t>(packet.body.size());
UpdateTransfer(transferId, offset, std::max(total, offset));
```

这三步分别表示：

1. 把这一块二进制数据写入文件
2. 更新当前已接收字节数
3. 更新服务端传输列表中的进度快照

这里也说明了为什么服务端能在管理面板上实时看到进度。

### 8. 服务端如何识别上传结束

如果收到的不是 `UploadData`，而是：

```cpp

```

说明客户端认为自己已经发完了。

这时服务端会：

1. `out.flush()`
2. 从结束包里读出 `sha256`
3. 对落盘后的真实文件重新计算 SHA-256

相关代码：

- `src/server/core.cpp:900`
- `src/server/core.cpp:906`

最终校验：

```cpp
const auto finalHash = Sha256File(resolved->real);
if (!expectedHash.empty() && finalHash != expectedHash) {
    ReplyError(sock, packet, "上传后校验失败");
    FinishTransfer(...);
    return;
}
```

这说明服务端不是仅仅信任客户端，而是自己重新算一次真正写入磁盘后的文件哈希。

### 9. 服务端如何返回上传成功

代码位置：

- `src/server/core.cpp:913`

返回：

```cpp
Cmd::UploadDone
SerializePairs({
    {"size", std::to_string(offset)},
    {"sha256", finalHash}
})
```

也就是说，服务端会把：

- 最终文件大小
- 最终文件哈希

发回给客户端。

客户端拿到之后，才会把整个上传任务判定为 `Success`。

## 四、哈希验证的完整逻辑要怎么讲

哈希验证不是“某一边顺手算了一下”，而是整个传输可靠性的一部分。

### 1. 哈希函数本身在哪里

核心函数：

- `Sha256File`：`src/shared/common.cpp:358`

它的实现特点是：

- 不是一次把整个文件读入内存
- 而是用 `64KB` 缓冲区流式读取
- 底层调用 Windows `BCrypt`

这点答辩时可以说：

> 我做的是流式哈希，不是整文件一次性读入内存，这样对大文件更稳妥。

### 2. 上传时哈希验证做了几次

上传至少做了两次：

#### 第一次：客户端上传前先算本地哈希

- `src/client/protocol.cpp:219`

目的：

- 先得到“正确文件哈希”

#### 第二次：服务端收完后重新算服务端文件哈希

- `src/server/core.cpp:906`

目的：

- 确保网络传输 + 磁盘写入后的文件内容是正确的

#### 第三次：客户端再比对服务端返回哈希

- `src/client/protocol.cpp:291`

目的：

- 双方对结果达成一致，避免服务端写坏但客户端误以为成功

### 3. 下载时哈希验证怎么做

下载逻辑在：

- 客户端：`DownloadFileSync`，`src/client/protocol.cpp:300`
- 服务端：`DownloadWorker`，`src/server/core.cpp:924`

服务端发送完成包时：

```cpp
SerializePairs({{"size", std::to_string(total)}, {"sha256", Sha256File(resolved->real)}})
```

代码位置：

- `src/server/core.cpp:995`

客户端下载完本地文件后：

```cpp
if (hashIt == result.end() || Sha256File(localPath) != hashIt->second) {
    return MakeResult(TransferOutcome::Failed, "下载后的哈希校验失败");
}
```

代码位置：

- `src/client/protocol.cpp:365`

所以下载也是“服务端算一次 + 客户端落盘后再算一次”。

### 4. 为什么不用 MD5，而用 SHA-256

答法建议：

> 因为这里的目标是做可靠性校验，SHA-256 的碰撞风险更低，也更符合现在的安全习惯。项目里直接使用了 Windows 自带的 BCrypt SHA-256 接口。

## 五、断点续传到底是怎么成立的

这一节是老师很喜欢追着问的重点。

### 1. 断点续传不是“记个进度条”，而是“协商偏移”

最本质的一句话：

> 断点续传成立的核心不是 UI，而是客户端和服务端就“从哪个字节开始继续”达成一致。

这个一致靠两个 `offset` 完成：

- 上传时：服务端返回 `offset`
- 下载时：客户端主动发送 `offset`

### 2. 上传续传的完整逻辑

#### 服务端侧

在 `UploadWorker` 中：

1. 先看目标文件是否存在
2. 如果存在，取当前文件大小作为 `offset`
3. 把 `offset` 发回客户端
4. 后续使用追加写模式继续写

关键代码：

- `src/server/core.cpp:847`
- `src/server/core.cpp:853`
- `src/server/core.cpp:864`

#### 客户端侧

在 `UploadFileSync` 中：

1. 收到 `UploadReady`
2. 解析出 `offset`
3. 用 `seekg(offset)` 跳过已经传过的部分
4. 只发送剩余数据块

关键代码：

- `src/client/protocol.cpp:249`
- `src/client/protocol.cpp:255`
- `src/client/protocol.cpp:260`

### 3. 下载续传的完整逻辑

#### 客户端侧先看本地已有多少

在 `DownloadFileSync` 中：

```cpp
std::uint64_t offset =
    std::filesystem::exists(localPath, ec)
        ? std::filesystem::file_size(localPath, ec)
        : 0;
```

代码位置：

- `src/client/protocol.cpp:306`

然后发给服务端：

```cpp
SerializePairs({
    {"path", remotePath},
    {"offset", std::to_string(offset)},
})
```

代码位置：

- `src/client/protocol.cpp:314`

#### 服务端侧从这个偏移开始读

在 `DownloadWorker` 中：

```cpp
std::uint64_t offset = pairs.contains("offset") ? ParseU64(pairs.at("offset")) : 0;
if (offset > total) {
    offset = 0;
}
in.seekg(static_cast<std::streamoff>(offset));
```

代码位置：

- `src/server/core.cpp:952`
- `src/server/core.cpp:965`

所以下载续传的核心是：

- 客户端告诉服务端“我这边已经有多少字节了”
- 服务端从那个位置继续发

### 4. 暂停后为什么还能继续？

因为暂停不是销毁进度，而是保留现有文件内容和已完成字节数。

上传暂停时：

- 已经写到服务端磁盘的部分会保留
- 下次继续时，服务端重新读目标文件长度，得到新的 `offset`

下载暂停时：

- 已经写到本地磁盘的部分会保留
- 下次继续时，客户端重新读本地文件长度，得到新的 `offset`

所以“继续”不是恢复某个线程现场，而是：

> 重新建连接，重新协商偏移，再从文件长度对应的位置继续。

这比保存线程上下文简单得多，也稳定得多。

### 5. 为什么取消后也可能还能续传？

因为取消逻辑只是停止当前连接，不一定删除已经传好的部分文件。

客户端任务层里，取消只会把状态改成：

- `Cancelled`

但是已经成功写入的本地或远程文件片段并不会自动清除。

所以只要部分文件还在：

- 上传可以继续
- 下载也可以继续

这也是断点续传能覆盖“失败后继续”“暂停后继续”“中断后继续”的原因。

你可以答：

> 哈希验证使用的是 SHA-256，公共实现函数在 `Sha256File`。上传时客户端先算本地文件哈希，服务端收完再算服务端文件哈希，并把最终哈希返回给客户端，客户端再比对一次。下载时则是服务端在发送结束包时带上源文件哈希，客户端下载落盘后重新算本地文件哈希再比对。这样保证不是只传完了，而是内容也一致。

### 问：你重点讲一下断点续传

你可以答：

> 断点续传的核心是偏移协商。上传时由服务端根据已存在目标文件长度返回 `offset`，客户端用 `seekg(offset)` 跳过已传部分继续发；下载时由客户端读取本地已存在文件长度作为 `offset` 发给服务端，服务端从该偏移 `seekg` 后继续发送剩余部分。暂停、失败或中断后之所以还能继续，本质上都是因为已经落盘的部分文件被保留了，下次只需要重新建连接并重新协商偏移。

## 七、老师继续深挖时你要抓住的三个关键词

如果你答辩现场紧张，记住这三个关键词就够了：

### 关键词 1：独立传输连接

> 上传下载不走控制连接，而是单独建 socket。

### 关键词 2：偏移协商

> 断点续传不是记忆线程状态，而是通过 `offset` 协商继续位置。

### 关键词 3：双端哈希校验

> 哈希不是单边算一次，而是客户端和服务端都参与最终一致性确认。
